Impressive Demos, Unreliable Products

When Google’s AI Leadership Asks for the Wrong Improvements – By Ed Dogan

Over the long weekend, I decided to give Google's AI toys another chance. They've been trying hard and had some impressive demos especially with "Nano banana". The test was of course against my favorite coder Claude. I gave them the same prompt — build a simple memory game app — and watched how they approached it.

The setup

The rules were simple:

  • Same prompt for both models.
  • Goal: Build a mobile memory game deployable to Google PlayStore.
  • Explicit instruction at the end: “If you have any questions, ask me first.”

Google AI Studio

Google’s assistant jumped straight into coding without asking a single clarifying question. It built the app in TypeScript, despite my instructions that it would need to be deployed to the Google Play Store.

  • Time Taken: ~2 minutes 11 seconds (including ~35 seconds of debugging)
  • Approach: Prioritized speed above everything else
  • Result: The game ran initially but crashed after finding the first pair

Claude

Claude, on the other hand, paused to ask four clarifying questions — even asking me what the game should be called. It then built the app with create-react-expo, a framework better suited for mobile deployment.

  • Time Taken: ~7 minutes 30 seconds to build, plus ~4 minutes 40 seconds to run & test (including waiting for my approvals)
  • Approach: Focused on intent alignment and usability
  • Result: The game worked almost perfectly, only minor bug was with animations blocking clicks.

The Tradeoff

In real-world engineering, choosing the right tradeoff is everything

I did not stop at Code Assistant and played with many of the impressive assistants they have. Speed was unmatched. For example, Nano Banana generated a complex game background asset in just 11 seconds. It wasn’t usable (looked like something from an old game), but it did follow the prompt. The pattern was clear: Google has thrown everything at speed.

This is not an accident — it’s a leadership decision. And it was glaring...

OpenAI has been heading down a similar path, pouring effort into shaving seconds off response times. But I don't think it's the way to lead the AI revolution.

Faster demos with unstable products is a recipe for failure. If we want AI to become a foundation of real-world software, leaders should be pushing for reliability and alignment over raw speed. Otherwise, the trust gap will swallow the hype.
